# Кнут и Пряник / Carrot & Stick — концепция продукта

## 1. Краткое описание

**Кнут и Пряник** — минималистичный трекер повторяющихся задач и привычек, построенный вокруг простой игровой механики: каждый день пользователь сам честно оценивает выполнение задачи и ставит ей **награду** или **наказание**.

Это не классический todo-лист и не сложный менеджер проектов. Это небольшой личный дневник самодисциплины, где пользователь не закрывает задачи, а каждый день отвечает на простой вопрос:

> Сегодня эта задача заслужила пряник, кнут или пока ничего?

Главная ценность продукта — простота, эмоциональность и быстрый ежедневный ритуал. Пользователь должен открыть приложение, за 10–15 секунд отметить нужные задачи и сразу увидеть итог своего поведения за выбранный период.

---

## 2. Философия

Приложение строится вокруг идеи **кнута и пряника**:

- **Пряник** — задача выполнена, пользователь молодец.
- **Кнут** — задача провалена, пользователь честно фиксирует неудачу.
- **Пустое состояние** — задача сегодня не оценивалась и не должна влиять на статистику.

Важно: приложение не наказывает пользователя автоматически. Оно не решает за него, что задача провалена. Пользователь сам выбирает, поставить кнут, пряник или оставить день пустым.

Это делает механику мягкой, но честной. Пользователь не получает бесконечные автоматические провалы за то, что не открыл приложение, но при этом может сам зафиксировать неудачу, если считает это правильным.

---

## 3. Продуктовая идея

Большинство todo-приложений отвечают на вопрос:

> Что нужно сделать?

**Кнут и Пряник** отвечает на другой вопрос:

> Как я справляюсь с тем, что считаю важным?

Поэтому основной объект приложения — не одноразовая задача, а **повторяющаяся задача / привычка**.

Примеры:

- Читать книгу.
- Учить английский.
- Ходить в зал.
- Не есть фастфуд.
- Писать диплом.
- Купить молоко, если пользователь хочет временно отслеживать это как задачу.

Даже одноразовую задачу пользователь может добавить, отметить кнутом или пряником, а затем архивировать.

---

## 4. Целевая аудитория

Приложение подходит людям, которым нужен простой личный трекер без перегруза:

- пользователям, которые хотят отслеживать привычки;
- людям, которым нравятся игровые механики;
- тем, кто устал от сложных todo-сервисов;
- тем, кто хочет видеть честную статистику своих действий;
- пользователям, которым важен быстрый мобильный сценарий.

---

## 5. Основные принципы продукта

### 5.1. Минимализм

В MVP не должно быть категорий, тегов, приоритетов, дедлайнов, kanban-досок и сложных сценариев.

Пользователь должен видеть:

- задачи;
- историю последних дней;
- быстрые кнопки награды и наказания;
- простую статистику.

### 5.2. Mobile-first

Интерфейс проектируется в первую очередь под мобильный экран.

Desktop-версия не должна превращаться в отдельный сложный интерфейс. На компьютере приложение должно выглядеть как аккуратная мобильная версия по центру экрана или в ограниченной по ширине колонке.

### 5.3. Единый интерфейс на всех устройствах

Пользователь должен получать одинаковый опыт на телефоне, планшете и компьютере.

Основная навигация — нижняя панель с иконками:

- задачи;
- статистика;
- аккаунт.

### 5.4. Игровая эмоциональность

Приложение должно ощущаться не как таблица, а как маленькая игра про самодисциплину.

Пользователь не просто закрывает задачу. Он получает пряник или кнут.

### 5.5. Отсутствие автоматического осуждения

Если пользователь ничего не выбрал, это не считается провалом.

`null` — нормальное третье состояние задачи за день. Оно не участвует в статистике.

---

## 6. Локализация и версии продукта

Приложение должно поддерживать две локализации:

- русский язык;
- английский язык.

При регистрации пользователь выбирает язык интерфейса.

Язык влияет на тексты приложения, но не должен жестко привязываться к набору иконок. При этом по умолчанию выбирается подходящий набор:

- для русского языка — **Пряник + Кнут**;
- для английского языка — **Carrot + Stick**.

После регистрации пользователь может изменить язык и набор иконок во вкладке аккаунта.

---

## 7. Наборы иконок

Emoji не подходят как основа визуального стиля, потому что на Windows, Android, iOS и других платформах они выглядят по-разному. Кроме того, стандартной надежной emoji именно для кнута нет.

Поэтому в MVP используются **предустановленные icon packs**.

### 7.1. Базовые наборы

#### Русский набор

- Награда: пряник / печенье.
- Наказание: кнут.

Возможный id набора:

```text
cookie_whip
```

#### Английский набор

- Reward: carrot.
- Punishment: stick.

Возможный id набора:

```text
carrot_stick
```

### 7.2. Технический принцип

Интерфейс не должен знать, что именно используется: emoji, SVG или другой ассет.

Компоненты должны работать с абстракцией:

```ts
type AppIcon = {
  type: "emoji" | "asset";
  value: string;
};
```

Пример набора:

```ts
const ICON_PACKS = {
  cookie_whip: {
    reward: {
      type: "asset",
      value: "/assets/iconpacks/cookie_whip/reward.svg"
    },
    punishment: {
      type: "asset",
      value: "/assets/iconpacks/cookie_whip/punishment.svg"
    }
  },
  carrot_stick: {
    reward: {
      type: "asset",
      value: "/assets/iconpacks/carrot_stick/reward.svg"
    },
    punishment: {
      type: "asset",
      value: "/assets/iconpacks/carrot_stick/punishment.svg"
    }
  }
};
```

Для MVP кастомная загрузка пользовательских иконок не нужна. Пользователь выбирает только из предустановленных наборов.

После MVP можно добавить возможность загружать свои reward/punishment иконки.

---

## 8. Главная страница

Главная страница — основной экран приложения.

### 8.1. Верхний блок статистики

Верхний блок показывает количество наград и наказаний за выбранный период.

Визуальная идея:

```text
      🍪  🪢
      15   4
[<] за неделю [>]
```

В реальной реализации вместо emoji могут отображаться SVG-иконки из выбранного icon pack.

### 8.2. Периоды на главной

Порядок переключения периодов на главной:

```text
За все время → За неделю → За день → За месяц → За год
```

Это сделано намеренно: пользователю интереснее сначала видеть накопленный результат и недельную динамику, а не только сегодняшний день.

### 8.3. Автоматическое переключение периода

На главной странице период может автоматически переключаться каждые 15 секунд.

Если пользователь вручную нажал стрелку, авто-переключение желательно временно поставить на паузу, чтобы интерфейс не раздражал и не спорил с пользователем.

### 8.4. Форматирование чисел на главной

На главной странице большие числа нужно сокращать до компактного формата, чтобы интерфейс не ломался.

Правило: показывать до трех значимых символов.

Примеры:

```text
10      → 10
100     → 100
1000    → 1,0К
5678    → 5,67К
11111   → 11,1К
100999  → 100К
1234567 → 1,23М
```

В статистике значения выводятся полностью.

## 9. Список задач

### 9.1. Выбор даты

Над списком задач находится выбор даты.

По умолчанию выбрана сегодняшняя дата.

Пользователь может выбрать прошлый день и поставить задаче пряник или кнут задним числом.

### 9.2. Карточка задачи

Карточка задачи должна быть простой и информативной.

Концепт:

```text
-------------------------------------------------
Название задачи для выполнения       [🍪]
История: — — 🍪 — 🪢 🍪 —            [🪢]
-------------------------------------------------
```

В реальном интерфейсе:

- слева отображается название задачи;
- под названием — история за последние 7 дней;
- справа две кнопки друг над другом: reward и punishment;
- выбранное состояние визуально подсвечивается;
- повторное нажатие на выбранную кнопку снимает оценку.

### 9.3. История за 7 дней

История отображает последние 7 дней существования задачи.

Если задача еще не существовала в один из дней, этот день не отображается.

Пример: задача создана 6 июня. На 10 июня история может выглядеть так:

```text
🍪 ○ 🍪 🪢 ○
```

А не так:

```text
○ ○ ○ ○ ○ 🍪 ○
```

Обозначения:

- reward icon — задача выполнена;
- punishment icon — задача провалена;
- пустая точка / `○` — состояние `null`, не учитывается в статистике.

---

## 10. Поведение задачи

### 10.1. Состояния оценки за день

У задачи на конкретную дату может быть одно из трех состояний:

```text
reward
punishment
null
```

`null` означает, что пользователь ничего не выбрал.

В базе данных `null` можно не хранить физически. Отсутствие записи за дату означает пустое состояние.

### 10.2. Переключение состояния

Если пользователь нажал reward:

- если раньше было `null` — становится `reward`;
- если раньше было `punishment` — становится `reward`;
- если раньше уже было `reward` — становится `null`.

Аналогично для punishment.

### 10.3. Просмотр деталей задачи

При нажатии на карточку открывается модальное окно.

В модальном окне отображается:

- название задачи;
- дата создания;
- общее количество reward;
- общее количество punishment;
- действия: редактировать, закрепить/открепить, архивировать.

---

## 11. Архивирование вместо удаления

В MVP задачи нельзя удалить навсегда.

Кнопка удаления фактически означает архивирование.

Архивная задача:

- не отображается на главной странице;
- остается в статистике;
- сохраняет всю историю;
- может быть восстановлена.

Внизу списка задач должна быть кнопка:

```text
Архивные задачи
```

или в английской версии:

```text
Archived tasks
```

На экране архивных задач пользователь может восстановить задачу.

Такой подход важен для честной статистики: прошлые кнуты и пряники не должны исчезать только потому, что пользователь убрал задачу с главной.

---

## 12. Закрепление задач

Для MVP поддерживается простое закрепление задач.

Сортировка:

1. Сначала закрепленные задачи.
2. Среди закрепленных выше находится та, которая была закреплена последней.
3. Затем обычные задачи в порядке создания.

Drag&Drop не входит в MVP, но может быть добавлен позже.

---

## 13. Статистика

Экран статистики показывает результаты за выбранный период.

### 13.1. Периоды статистики

Порядок периодов на вкладке статистики:

```text
За день → За неделю → За месяц → За год → За все время
```

По умолчанию выбран период:

```text
За неделю
```

Пользователь может переключаться на предыдущие и следующие периоды.

Для периода `За все время` стрелки переключения прошлых/следующих периодов не нужны.

### 13.2. Неделя

В MVP неделя считается с понедельника по воскресенье.

После MVP можно добавить настройку первого дня недели.

### 13.3. Содержимое статистики

На экране статистики отображается:

1. Выбор периода.
2. Навигация по периодам.
3. Общее количество reward и punishment за период.
4. Список задач за период.

Для каждой задачи:

```text
Название задачи
Пряники: 15
Кнуты: 7
```

В MVP проценты, графики и дополнительные коэффициенты не показываются.

### 13.4. Какие задачи показывать

Если задача на начало периода еще не существовала, она не показывается.

Если задача уже существовала, но за период у нее 0 reward и 0 punishment, она показывается с нулевыми значениями.

Архивные задачи тоже должны участвовать в статистике, если они существовали в выбранный период.

---

## 14. Аккаунт

Во вкладке аккаунта пользователь может:

- увидеть свой логин;
- изменить язык интерфейса;
- выбрать набор иконок;
- изменить пароль;
- выйти из аккаунта.

Восстановление пароля в MVP не предусмотрено.

При регистрации нужно явно предупредить пользователя, что восстановить пароль нельзя.

---

## 15. Регистрация и авторизация

Для MVP достаточно:

- логин;
- пароль.

Email не нужен.

У каждого пользователя свои задачи, свои оценки, свои настройки и своя статистика.

При регистрации пользователь выбирает:

- язык интерфейса: русский или английский;
- набор иконок можно выбрать автоматически на основе языка.

---

## 16. MVP

В MVP входит:

- регистрация по логину и паролю;
- вход;
- выход;
- смена пароля;
- выбор языка;
- выбор icon pack;
- создание повторяющихся задач;
- редактирование задач;
- архивирование задач;
- восстановление архивных задач;
- закрепление задач;
- оценка задачи за выбранную дату: reward / punishment / null;
- история задачи за последние 7 дней;
- главная статистика с периодами;
- вкладка статистики с периодами;
- mobile-first интерфейс;
- Docker Compose для локального запуска;
- возможность открыть приложение с телефона в локальной сети.

---

## 17. Не входит в MVP

В MVP не нужно делать:

- email;
- восстановление пароля;
- OAuth;
- категории;
- теги;
- дедлайны;
- напоминания;
- push-уведомления;
- загрузку пользовательских иконок;
- drag&drop сортировку;
- графики;
- проценты выполнения;
- публичные профили;
- социальные функции;
- экспорт статистики;
- темную тему, если она заметно задерживает разработку.

---

## 19. Тон интерфейса

Интерфейс должен быть дружелюбным, легким и немного игровым.

Он не должен звучать как корпоративный таск-менеджер.

Примеры хорошего тона:

- «Заслужил пряник?»
- «Получил кнут?»
- «Сегодня пока без оценки».
- «Архивировать задачу».
- «Вернуть задачу из архива».

Но важно не переборщить. Основной интерфейс должен оставаться понятным и спокойным.

---

## 20. Главная идея, которую нельзя потерять

**Кнут и Пряник** — это не менеджер задач. Это дневник самодисциплины.

Пользователь не закрывает задачи. Он ежедневно выносит себе честную оценку: заслужил сегодня пряник или получил кнут.

Именно эта простая механика делает приложение эмоциональным, понятным и мотивирует возвращаться каждый день.
